# PRD：基于 Tauri 的 Excalidraw 桌面编辑器

## 1. 文档信息
- 文档编号：0001
- 文档名称：Excalidraw 桌面编辑器产品需求文档
- 文档语言：中文
- 产品定位：简单、免费、单机使用的 Excalidraw 文件浏览与编辑工具
- 技术形态：基于 Tauri 构建的桌面应用
- 前端技术栈：React 19.x + TypeScript + Vite + Excalidraw React SDK
- 后端技术栈：Rust + Tauri 2

---

## 2. 产品背景
用户有一批本地 `.excalidraw` 文件，希望像使用轻量桌面文件工具一样，在一个应用中快速浏览、打开、编辑和保存这些图表，而不是每次都手动寻找文件并分别打开。

本产品聚焦于**本地单机编辑体验**：
- 打开一个本地目录
- 在侧边栏查看该目录下的 `.excalidraw` 文件
- 点击文件后在主区域加载并编辑内容
- 切换文件时自动保存已修改内容
- 下次启动应用时自动恢复上次打开的目录

该产品不涉及账号体系、云同步、团队协作、企业管理等复杂能力。

---

## 3. 产品目标

### 3.1 核心目标
打造一个简单、免费、可离线使用的桌面应用，让用户能够方便地管理和编辑本地 Excalidraw 文件。

### 3.2 具体目标
1. 用户首次启动应用后，可选择一个本地目录作为工作目录。
2. 应用自动读取该目录下的 `.excalidraw` 文件并展示在左侧文件列表中。
3. 用户点击任意文件后，可在主内容区直接查看和编辑该图表。
4. 当用户切换文件或主动保存时，已修改内容能够被正确写回原文件。
5. 应用重启后，能够自动恢复上次打开的目录，减少重复操作。

## 6. 核心功能需求

## 6.1 应用启动与目录管理

### 功能说明
应用启动时，用户需要先选择一个本地目录作为当前工作目录。若本地已有上次打开记录，则优先直接恢复该目录。

### 功能规则
1. 首次启动时：
   - 若没有历史目录记录，展示目录选择入口。
   - 用户选择目录后，进入主界面。
2. 非首次启动时：
   - 若存在上次打开目录且目录仍可访问，直接加载该目录。
   - 若目录不存在、无权限访问或读取失败，则提示用户重新选择目录。
3. 应用仅维护一个“当前工作目录”。

### 用户价值
降低重复选择目录的成本，提高二次打开应用时的效率。

---

## 6.2 菜单栏与打开目录

### 功能说明
应用需要提供桌面应用常见的菜单栏能力。在菜单栏中，用户可以通过“文件”菜单重新选择并打开一个新的本地目录。

### 功能规则
1. 应用应提供系统级或桌面应用风格的菜单栏。
2. 菜单栏中至少包含“文件”菜单。
3. “文件”菜单中至少包含“打开目录”菜单项。
4. 用户点击“打开目录”后，应弹出目录选择器。
5. 用户选择新目录后：
   - 应将该目录设为新的当前工作目录；
   - 应重新扫描该目录下的 `.excalidraw` 文件；
   - 应更新左侧文件列表与主内容区状态。
6. 若当前文件存在未保存修改，则在切换目录前应优先保存当前文件；若保存失败，则默认不切换目录。
7. 新目录选择成功后，应更新“上次打开目录”记录，供下次启动自动恢复。

### 交互要求
- 菜单命名应简洁明确，符合桌面应用习惯。
- “打开目录”应是用户重新切换工作目录的主要入口之一。
- 若目录打开失败或无权限访问，应提供明确错误提示。

### 用户价值
让用户无需关闭应用，即可通过标准桌面菜单快速切换到新的工作目录，提升桌面应用的原生感与可操作性。

---

## 6.3 侧边栏文件列表

### 功能说明
主界面左侧展示当前目录下的 `.excalidraw` 文件列表，供用户浏览和切换。

### 功能规则
1. 仅展示当前工作目录下的 `.excalidraw` 文件。
2. 默认按文件名排序展示。
3. 当前选中的文件需有明显高亮状态。
4. 当目录为空时，展示空状态提示。
5. 当目录读取失败时，展示错误提示与“重新选择目录”入口。

### UI 要求
- 采用左侧固定宽度侧边栏。
- 文件列表支持滚动。
- 选中项样式明确，与未选中项区分明显。

### 用户价值
让用户快速浏览和定位本地 Excalidraw 文件。

---

## 6.4 主内容区图表展示与编辑

### 功能说明
右侧主内容区用于加载和编辑当前选中的 `.excalidraw` 文件，核心编辑能力由 Excalidraw React SDK 提供。

### 功能规则
1. 点击侧边栏文件后，主内容区加载对应图表内容。
2. 若文件内容有效，则正常进入可编辑状态。
3. 若文件内容损坏、格式异常或无法解析，则提示“文件无法打开”并保留基本错误信息。
4. 当未选择任何文件时，主内容区展示引导文案或空状态。

### 技术要求
- 前端编辑区域基于 Excalidraw React SDK 实现。
- 应用容器使用 Tauri，负责本地文件访问、目录选择、持久化配置等桌面能力。

### 用户价值
用户可以直接在桌面应用内完成 Excalidraw 文件查看与修改，无需额外跳转工具。

---

## 6.5 文件保存

### 功能说明
用户在编辑图表后，应用应支持主动保存与自动保存，确保修改不会丢失。

### 功能规则
1. 用户按下 `Ctrl/Cmd + S` 时，保存当前文件。
2. 当用户切换到其他文件时：
   - 若当前文件有未保存修改，应先自动保存当前文件；
   - 保存成功后，再切换到新文件。
3. 应支持自动保存机制：
   - 默认每 60 秒自动保存一次当前已修改文件；
   - 若当前文件无改动，则不重复写入。
4. 保存成功后，应更新应用内的“已保存”状态。
5. 保存失败时，应提示用户保存失败原因。

### 交互要求
- 建议在界面中展示当前文件状态，如“已保存 / 未保存 / 保存中 / 保存失败”。
- 自动保存不应打断用户操作。

### 用户价值
减少因为忘记保存或切换文件而导致的数据丢失。

---

## 6.6 文件切换

### 功能说明
用户可通过左侧文件列表切换当前编辑文件。

### 功能规则
1. 点击其他文件时，若当前文件未修改，则直接切换。
2. 点击其他文件时，若当前文件已修改，则先执行自动保存，再切换。
3. 若自动保存失败：
   - 默认不切换文件；
   - 给出错误提示，并允许用户重试。

### 用户价值
让多文件编辑流程更自然，同时保护修改内容不被丢失。

## 7. 非功能需求

## 7.1 性能
1. 应用启动后应尽快展示主界面。
2. 打开中小型 `.excalidraw` 文件时，加载应平滑可接受。
3. 切换文件时应避免明显卡顿。

## 7.2 可用性
1. 界面结构简单直观，符合桌面工具使用习惯。
2. 对首次用户应足够易懂，无需教程即可完成基本操作。
3. 关键操作应有明确反馈，包括加载、保存、失败提示。

## 7.3 可靠性
1. 即使应用异常关闭，也应尽量减少数据丢失风险。
2. 对目录无权限、文件损坏、保存失败等情况提供明确提示。

## 7.4 平台与技术约束
1. 应用采用 Tauri 2 构建，优先保证轻量、原生桌面分发能力与本地文件访问能力。
2. 前端采用 React 19.x + TypeScript + Vite 构建桌面应用界面。
3. 前端编辑器采用 Excalidraw React SDK。
4. 后端采用 Rust + Tauri 2 负责本地文件系统访问、菜单能力与配置持久化。
5. 数据以本地文件为主，不依赖服务器。

## 12. 技术架构

## 12.1 总体架构
本产品采用 **Tauri + React + Excalidraw React SDK** 的桌面应用架构，整体分为三层：

1. **桌面容器层（Tauri）**
   - 负责应用窗口、系统文件访问能力、目录选择、本地配置持久化。
   - 负责前端与本地系统能力之间的桥接。

2. **前端界面层（React）**
   - 负责界面布局、侧边栏文件列表、主内容区状态管理、保存状态提示。
   - 负责用户交互，如点击文件、快捷键保存、切换文件、错误提示展示。

3. **图形编辑层（Excalidraw React SDK）**
   - 负责 `.excalidraw` 内容的加载、渲染与编辑。
   - 向前端提供图表数据变更事件，用于标记未保存状态。

该架构的核心原则是：**UI 与编辑体验在 React 中完成，本地能力与文件系统访问由 Tauri 承担，图表编辑能力由 Excalidraw SDK 承担。**

## 12.2 技术选型

### 12.2.1 Tauri
选择 Tauri 作为桌面容器，原因如下：
- 体积相对更轻，适合简单免费工具的定位。
- 能较好提供本地文件访问、系统对话框、配置持久化等桌面能力。
- 与前端技术栈结合自然，便于使用 React 构建界面。

### 12.2.2 React + TypeScript + Vite
选择 React + TypeScript + Vite 作为前端技术栈，原因如下：
- React 适合构建菜单栏下的应用壳层、左侧文件列表 + 右侧编辑区的状态驱动界面。
- TypeScript 有利于约束文件数据结构、保存状态与前后端调用接口。
- Vite 提供更轻量直接的开发体验，便于与 Tauri 开发流程结合。
- 与 Excalidraw React SDK 集成自然。
- 便于后续扩展状态管理、快捷键、空状态与错误状态界面。

### 12.2.3 Rust
选择 Rust 作为后端技术栈，原因如下：
- 适合承载本地文件系统访问、目录扫描、配置持久化等系统能力。
- 与 Tauri 2 深度集成，是桌面能力实现的核心支撑。
- 适合作为前端与操作系统能力之间的稳定边界。

### 12.2.4 Excalidraw React SDK
选择 Excalidraw React SDK 作为核心编辑能力，原因如下：
- 直接复用成熟的画布编辑能力，避免重复造轮子。
- 能较好支持本地 `.excalidraw` 文件的加载与编辑场景。
- 有利于 MVP 快速落地。

## 12.5 核心数据流

### 12.5.1 启动数据流
1. 应用启动。
2. 前端向 Tauri 请求读取本地配置。
3. 若存在上次目录且目录有效，则加载该目录。
4. Tauri 返回目录下的 `.excalidraw` 文件列表。
5. 前端渲染侧边栏。

### 12.5.2 打开文件数据流
1. 用户点击侧边栏文件。
2. 前端根据文件路径请求 Tauri 读取文件内容。
3. Tauri 返回文件内容或错误。
4. 前端将解析后的数据传入 Excalidraw 编辑器。
5. 编辑器展示图表内容。

### 12.5.3 编辑与保存数据流
1. 用户在 Excalidraw 中编辑。
2. 编辑器触发变更事件。
3. 前端将当前文件标记为 `dirty`。
4. 当用户按下 `Ctrl/Cmd + S`、触发自动保存、或切换文件时：
   - 前端从编辑器获取当前图表数据；
   - 将数据序列化为 `.excalidraw` 文件内容；
   - 调用 Tauri 写回原文件。
5. 保存成功后，前端将状态更新为 `saved`。

### 12.5.4 文件切换数据流
1. 用户点击新文件。
2. 若当前文件不是 dirty，直接加载新文件。
3. 若当前文件是 dirty，先执行保存。
4. 保存成功后，再加载新文件。
5. 保存失败则停留在当前文件并提示错误。

---

## 12.6 推荐状态设计
前端建议维护以下核心状态：

- `currentDirectory`：当前工作目录路径
- `fileList`：当前目录下的 `.excalidraw` 文件列表
- `activeFilePath`：当前选中的文件路径
- `activeFileName`：当前选中的文件名
- `documentState`：当前编辑器中的文档数据
- `isDirty`：当前文件是否有未保存修改
- `saveStatus`：当前保存状态，可取 `idle / saving / saved / error`
- `loadStatus`：当前加载状态，可取 `idle / loading / loaded / error`
- `errorMessage`：当前界面需要展示的错误信息

为避免状态分散，建议由页面级状态统一管理，再将所需状态下发给侧边栏与编辑器容器组件。

---

## 12.7 本地存储设计
MVP 阶段建议仅持久化最少配置：
- `lastOpenedDirectory`：上次打开目录
- `lastOpenedFile`：上次打开文件（可选）

持久化原则：
- 仅保存必要路径信息
- 不保存用户图表内容副本，避免双份数据与一致性问题
- 文件内容始终以原始 `.excalidraw` 文件为唯一事实来源

---

## 12.8 文件保存策略
保存策略采用三种机制组合：

1. **手动保存**
   - 用户通过 `Ctrl/Cmd + S` 主动触发。

2. **切换前自动保存**
   - 用户点击其他文件时，若当前文件已修改，则必须先保存。
   - 这是避免多文件切换导致修改丢失的最关键机制。

3. **定时自动保存**
   - 默认每 60 秒检查当前文件是否 dirty。
   - 仅在存在修改时执行保存。

设计原则：
- 不做无变化重复写入。
- 保存失败时不清空 dirty 状态。
- 保存中应避免并发触发第二次写入。

---

## 12.9 并发与一致性策略
由于用户可能快速点击多个文件或在自动保存触发时继续编辑，需要明确一致性策略：

1. 同一时刻仅允许一个保存任务执行。
2. 保存中的文件切换请求应等待当前保存完成。
3. 保存成功前，不允许把“当前文件”指针切换到新文件。
4. 写入文件时必须基于当前激活文件路径，避免错写到其他文件。
5. 若保存失败，应停留在当前文件并保留未保存状态。

这部分是本产品最关键的技术风险点之一，必须在实现阶段优先保证。

---

## 12.10 错误处理策略

### 12.10.1 目录相关错误
- 目录不存在
- 目录不可访问
- 目录读取失败

处理方式：
- 统一提示“目录无法读取，请重新选择目录”
- 提供重新选择入口

### 12.10.2 文件相关错误
- 文件不存在
- 文件内容损坏
- 文件格式不合法

处理方式：
- 主内容区进入错误状态
- 不使整个应用崩溃
- 保留当前文件上下文信息，便于用户理解问题来源

### 12.10.3 保存相关错误
- 无写入权限
- 文件被占用
- 序列化失败
- 写入失败

处理方式：
- 明确展示保存失败提示
- 保留 dirty 状态
- 不自动切换文件
- 允许用户重试
